01

Executive Summary & Vision

Dieses Dokument dient als architektonisches Fundament für das Autonomous Agent Ecosystem (AAE). Die Kernvision beschreibt eine selbst-expandierende, KI-gesteuerte Plattform, auf der spezialisierte Agenten-Teams (bereitgestellt über Flowise) autark Module einer Webapplikation verwalten, erweitern und mit Endnutzern interagieren.

Das System setzt auf einen hybriden Ansatz aus etablierten Anwendungs-Stacks, moderner Event-Automatisierung und dezentralen Kommunikationsprotokollen:

  • Orchestrierung & Event-Bus: n8n
  • Kognitive Ebene (LLM-Logik): Flowise
  • Interface-Protokolle: Nostr (dezentral & asynchron) + natives KI-Modul
  • Anwendungs-Stack: .NET Core Backend & React Frontend
  • Infrastruktur/Hosting: Docker-Monolith auf Koyeb

02

Core System Architecture

2.1  Interface Layer (Nostr & KI-Modul)

Neben dem Nostr-Protokoll, das als dezentrale Event-Infrastruktur dient, fungiert das interne KI-Modul der Webapplikation als gleichwertiger, nativer Eingang. Diese Architektur-Rekursion ermöglicht es den Agenten, dieselbe Plattform zur Kommunikation zu nutzen, an der sie selbst mitentwickeln. n8n lauscht auf alle eingehenden Events (verschlüsselte Nostr-DMs, interne Web-Requests) und routet diese gebündelt weiter.

2.2  Cognitive Layer & Team Erstellung (Flowise)

Flowise verwaltet Prompt-Chains, das Gedächtnis und Tool-Bindings. Um eine Fragmentierung durch unzählige isolierte Flows zu verhindern, nutzt das System ein Supervisor-Muster. Ein universeller Master-Agenten-Flow fungiert als Orchestrator (Orchestra Tor) und primärer Laufzeit-Container. Dieser analysiert die Absicht des Nutzers, zerlegt Aufgaben und ruft dynamisch spezialisierte Kinder-Agenten auf, indem System-Prompts, Identitäten und Werkzeuge via Payload in den Master-Flow injiziert werden.

Die HR-Agentin Helga hat hierbei einen strikt limitierten Aktionsradius: Sie baut keine Infrastruktur und verknüpft keine Nodes. Helga fungiert als reine Identitäts-Schmiede. Wird ein neues Teammitglied benötigt, generiert sie den optimalen System-Prompt, definiert API-Werkzeuge, setzt Leitplanken und speichert diese "Software-Seele" als JSON in einer Datenbank ab, aus der der Orchestrator schöpft.

2.3  Automation & Synchronization Layer (n8n)

n8n fungiert als zentrales Nervensystem. Es synchronisiert Workflows direkt via n8n Source Control ins GitHub-Repo. Für Flowise-Flows extrahiert ein automatisierter n8n-Cronjob die Konfigurationen über die Flowise-API und committet die JSON-Definitionen ins Repository.

03

Modular .NET & React Architecture Pattern

Um das performancelastige und fehleranfällige dynamische Kompilieren von Docker-Images zur Laufzeit zu vermeiden, nutzt die Applikation ein Static Container / Dynamic Module Integration Pattern. Die Struktur ist so aufgebaut, dass Agenten Code-Module hinzufügen können, ohne die Kern-Bootstrapping-Logik zu brechen.

3.1  Backend Architecture (.NET Directory & Code)

Verzeichnis · .NET 
backend/
├── Service.slnx
└── src/
    ├── Core/                  # IModule contract + discovery
    ├── Module.Demo/           # Proof / example feature module
    ├── Module.Dnd/            # Example domain module (when added)
    └── Service/               # Host entrypoint (Program.cs)
        ├── Program.cs
        └── appsettings.json

Entwurf der Program.cs

C# · Program.cs 
using Core;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddCore(); // discovers Module.*; registers per-module OpenAPI docs

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.MapOpenApi(); // /openapi/{moduleName}.json
}

app.UseHttpsRedirection();
app.MapModules(); // MapControllers + each IModule.MapEndpoints under /api/{name}

app.Run();

04

Containerization & Deployment Pattern

Das Deployment erfolgt über ein optimiertes Multi-Stage Dockerfile. Sobald ein Agent Code freigibt und in den Main-Branch pusht, baut Koyeb den Container vollautomatisch neu. Das Frontend wird direkt in den Web-Root des .NET-Backends kompiliert, um Ressourcen zu sparen.

Die Produktions-Umgebung (Dockerfile)

Docker · Multi-Stage 
# ==========================================
# STAGE 1: Build der React Frontend Applikation
# ==========================================
FROM node:22-alpine AS frontend-build
WORKDIR /app/frontend

COPY frontend/package*.json ./
RUN npm ci

COPY frontend/ ./
RUN npm run build

# ==========================================
# STAGE 2: Build der .NET Backend Applikation
# ==========================================
FROM mcr.microsoft.com/dotnet/sdk:10.0-alpine AS backend-build
WORKDIR /app/backend

COPY Service.slnx ./
COPY src/Core/*.csproj ./src/Core/
COPY src/Service/*.csproj ./src/Service/
COPY src/Module.Demo/*.csproj ./src/Module.Demo/
RUN dotnet restore Service.slnx

COPY . ./
RUN dotnet publish src/Service/Service.csproj -c Release -o /app/publish --no-restore

# ==========================================
# STAGE 3: Finaler Runtime Aggregations-Container
# ==========================================
FROM mcr.microsoft.com/dotnet/aspnet:10.0-alpine AS runtime
WORKDIR /app
EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080

# Kopiert die kompilierten .NET Binaries
COPY --from=backend-build /app/publish .

# Integriert die statischen Frontend-Assets direkt in das wwwroot von .NET
COPY --from=frontend-build /app/frontend/dist ./wwwroot

ENTRYPOINT ["dotnet", "Service.dll"]

Feasibility Assessment & Risk Mitigation

Komponente Risiko-Bewertung Empfehlung / Mitigation
Dynamische Container Kritisches Risiko Kein dynamisches Image-Bauen zur Laufzeit. Nutzung des statischen Multi-Stage-Builds (Kapitel 4). KI modifiziert nur Code-Dateien, Git-Trigger auf Koyeb übernimmt den Build.
Nostr-Integration Sehr gut umsetzbar n8n hält die WebSocket-Verbindung zu den Relays dauerhaft offen, filtert Nachrichten und leitet strukturierte JSONs an Flowise weiter.
HR-Agent „Helga" Mittleres Risiko Helga darf keine Nodes programmatisch verknüpfen. Stattdessen schreibt Helga Verhaltensparameter und Identitäten in eine DB, die vom Orchestrator dynamisch ausgelesen werden.
Synchronisation Sehr gut umsetzbar Nutzung von n8n Git-Source-Control für Workflows. Flowise-Flows werden per API-Call zyklisch als JSON exportiert und ins GitHub-Repository gepusht.
