01

Executive Summary & Vision

This document is the architectural foundation for the Autonomous Agent Ecosystem (AAE). Specialized agent teams build modular product features under human approval: Leo orchestrates, Helga forges identities, domain supervisors plan (and may nest further supervisors), and specialists implement inside module boundaries.

Hybrid stack in use today:

  • Orchestration & event bus: n8n (four stateless *-think workflows)
  • Optional LLM canvas: Flowise
  • Interfaces: Nostr (async DMs) + native React / KI UI on the web app
  • Application stack: ASP.NET Core (net10.0) + React (Vite)
  • Infrastructure: Docker images under infrastructure/, hosted on Koyeb

Core System Architecture

2.1  Interface layer (Nostr & KI UI)

Nostr is the decentralized event ingress. The web app’s KI / team UI is a first-class peer channel. Agents can use the same product they help build. n8n consumes relay traffic and app events and wakes the matching think webhook.

2.2  Agent hierarchy (recursive supervisors)

Runtime cognition for Leo, Helga, supervisors, and specialists runs in the four n8n think workflows (leo-think, helga-think, supervisor-think, specialist-think). Each wake is stateless: the backend injects context; the workflow calls back into https://ai.neberg.de/api/agents/….

Helga is identity-only: she does not wire workflows or write application code. She fills supervisor / specialist templates and posts create-identity. Supervisors may nest: a supervisor can request further supervisors under itself (managerId = parent supervisor-*) as well as specialists.

2.3  Automation layer (n8n)

n8n is the event bus and execution layer for the think workflows. Approvals are not n8n Wait nodes; HITL is owned by backend routes under Module.Agents (see process doc docs/process/human-in-the-loop.md). Workflow JSON lives in agents/n8n-workflows/.

Modular .NET & React Pattern

Static container / dynamic module integration: agents add Module.* projects without rewriting core bootstrap. Discovery registers each module under /api/{module.Name}.

04

Containerization & Deployment

The combined webapp image is a multi-stage Dockerfile under infrastructure/webapp/. Build context is the repository root. Frontend assets land in ASP.NET wwwroot.

Production Dockerfile (excerpt)

Docker · Multi-stage 
# STAGE 1: React frontend
FROM node:22-alpine AS frontend-build
WORKDIR /app/frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build

# STAGE 2: .NET backend (context = repo root)
FROM mcr.microsoft.com/dotnet/sdk:10.0-alpine AS backend-build
WORKDIR /app/backend
COPY backend/Service.slnx ./
COPY backend/src/Core/*.csproj ./src/Core/
COPY backend/src/Service/*.csproj ./src/Service/
COPY backend/src/Module.Agents/*.csproj ./src/Module.Agents/
COPY backend/src/Module.Demo/*.csproj ./src/Module.Demo/
RUN dotnet restore Service.slnx
COPY backend/ ./
RUN dotnet publish src/Service/Service.csproj -c Release -o /app/publish --no-restore

# STAGE 3: Runtime
FROM mcr.microsoft.com/dotnet/aspnet:10.0-alpine AS runtime
WORKDIR /app
EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080
COPY --from=backend-build /app/publish .
COPY --from=frontend-build /app/frontend/dist ./wwwroot
ENTRYPOINT ["dotnet", "Service.dll"]

Feasibility & Risk Mitigation

Component Risk Mitigation
Dynamic containers Critical No runtime image builds. Static multi-stage webapp image; agents change source only; platform rebuilds on push.
Nostr integration Good Dedicated relay wss://nostr.neberg.de; n8n Nostr nodes + Module.Agents listeners.
Helga (HR) Medium Identity forge only — templates + create-identity; no workflow wiring.
HITL approvals Medium Routes mapped on Module.Agents; handlers still stubbed (NotImplementedException). n8n stays Wait-free.
Workflow sync Good Version think workflows under agents/n8n-workflows/; import on n8n.neberg.de.
